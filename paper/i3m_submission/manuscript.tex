\author[a]{TBD Author}

\begin{abstract}
Financial event records used in downstream simulation workflows need explicit evidence links, update histories, and reproducible replay traces. This paper presents a reproducible versioned event graph simulation prototype for evidence-constrained financial event streams. The prototype accepts deterministic event records, checks required schema fields and exact evidence-span containment, applies a compact set of graph update operators, and replays controlled perturbation streams as a discrete-event simulation. The implemented operators add event nodes, merge duplicate records, update slots, mark unresolved conflicts, and append version logs. To strengthen the deterministic replay evaluation, we add two additional checks: negative-control cases that exercise schema, evidence, date, and unsupported-operator gates, and a scale-sensitivity run over controlled streams. The evaluation is intentionally limited to framework behavior: schema validity, evidence coverage, operator agreement on injected perturbations, replay completeness, and lightweight runtime behavior. The results support a conservative design-science claim: the implemented prototype can make controlled financial event graph updates reproducible and auditable without making extraction-performance, market-outcome, or decision-system claims.
\end{abstract}

\section{Introduction}

Financial event extraction and reporting pipelines often stop at structured records. For simulation and audit, however, the record alone is not enough: the system also needs evidence links, update histories, and a replayable state transition trace. A duplicate record, a revised amount, or a contradictory disclosure should leave an explicit graph transition rather than a silent overwrite.

This paper studies that narrower infrastructure problem. The contribution is a reproducible simulation-oriented prototype, not a financial event extraction benchmark system and not a financial prediction system. We ask whether a controlled stream of financial event records can be checked against source evidence, inserted into a versioned event graph, updated through a small deterministic operator set, and replayed as a discrete-event simulation with auditable artifacts.

The prototype follows a design-science framing. It specifies a compact event-record schema, an evidence gate based on exact span containment, a versioned event graph, and a replay simulator. The experiments use controlled perturbation streams to exercise duplicate merges, slot updates, conflict marks, and temporal replay order. The paper reports only numbers generated by the repository scripts.

\section{Related work}

\subsection{Financial event extraction and event records}

Document-level financial event extraction systems such as DCFEE \cite{yang2018dcfee} and Doc2EDAG \cite{zheng2019doc2edag} motivate the need for structured event records whose arguments may be distributed across a document. Their main problem is extraction from financial text. Our problem begins after event-like records are available: we study whether records can be evidence-constrained, versioned, and replayed reproducibly.

\subsection{Provenance, event sourcing, and versioned graph maintenance}

The W3C PROV-O vocabulary \cite{w3c2013provo} emphasizes explicit provenance over entities and activities. Event sourcing \cite{fowler2005eventsourcing} emphasizes reconstructing state from an event log. The prototype follows the same auditability principle in a smaller setting: event nodes, entity links, conflicts, merges, and version logs are all kept as replay artifacts.

\subsection{Discrete-event simulation for auditable event streams}

Discrete-event simulation represents system evolution as event-triggered state transitions \cite{banks2010discrete}. In this paper the event stream is not a market model and does not estimate future outcomes. The simulation role is to replay controlled record perturbations and expose each graph state transition for inspection.

\section{Problem definition and simulation model}

The input consists of deterministic seed records and a controlled perturbation stream. A seed record contains an event identifier, event type, subject, object, date, trigger, evidence span, source document identifier, source text, amount, and status. A stream record wraps an event payload with replay metadata such as arrival index, perturbation type, base event identifier, and expected operator.

At replay step $t$, the prototype maintains a versioned event graph
\begin{equation}
G_t = (V_t, E_t, L_t),
\end{equation}
where $V_t$ contains event and entity nodes, $E_t$ contains event-entity links, and $L_t$ contains append-only version logs. A replay operator $o_t$ induces the graph transition
\begin{equation}
G_t \xrightarrow{o_t} G_{t+1}.
\end{equation}

The simulation state is summarized as
\begin{equation}
S_t = \{g_t, n^e_t, n^v_t, n^m_t, n^u_t, n^c_t, n^{uc}_t, r_t\},
\end{equation}
where $g_t$ is the graph version, $n^e_t$ is the active event count, $n^v_t$ is the active entity count, $n^m_t$ is the merge count, $n^u_t$ is the update count, $n^c_t$ is the conflict count, $n^{uc}_t$ is the unresolved conflict count, and $r_t$ is the replay step. Each accepted record therefore induces
\begin{equation}
S_t \rightarrow S_{t+1}.
\end{equation}

\section{Prototype design}

\subsection{Evidence-constrained event records}

The schema module normalizes both flat seed records and nested stream records. A record is schema-valid only when required fields are present and the time field is an ISO date string. The evidence module then checks that the evidence span is non-empty and appears in the source text. The gate is intentionally exact and local; it does not infer evidence or call external services.

\subsection{Versioned event graph}

The graph store uses dictionary-backed event and entity nodes plus list-backed edges, conflicts, merges, and version logs. The graph version is the length of the version log. This simple representation is sufficient for a reproducible prototype because every accepted replay step appends an auditable record.

\subsection{Update operators}

The operator set is
\begin{equation}
\mathcal{O} = \{\texttt{ADD\_EVENT}, \texttt{MERGE\_EVENT}, \texttt{UPDATE\_SLOT}, \texttt{MARK\_CONFLICT}, \texttt{VERSION\_LOG}\}.
\end{equation}
\texttt{ADD\_EVENT} inserts a new event node and entity links. \texttt{MERGE\_EVENT} records that an incoming duplicate refers to an existing event. \texttt{UPDATE\_SLOT} records changed fields and updates the target event. \texttt{MARK\_CONFLICT} appends an unresolved conflict mark. \texttt{VERSION\_LOG} records the applied operation.

\subsection{Discrete-event replay simulation}

Replay orders records by arrival index. At each step the simulator runs schema validation, evidence matching, operator prediction, target selection, graph update execution, and state logging. \section{Experimental setup}

All experiments are local or server-side script runs with fixed seeds. They do not train models, run GPU inference, call external APIs, or copy model files. The main controlled stream starts from 30 seed samples and injects duplicate, conflict, update, and temporal replay perturbations. Negative controls add invalid or unsupported records to test gates. Scale sensitivity increases the seed count while keeping perturbations deterministic and proportional.

The metrics are defined as follows:
\begin{align}
\text{Schema Validity Rate} &= \frac{\text{valid records}}{\text{total records}},\\
\text{Evidence Coverage Rate} &= \frac{\text{records with matched evidence}}{\text{total records}},\\
\text{Merge Accuracy} &= \frac{\text{correct duplicate merges}}{\text{injected duplicates}},\\
\text{Conflict Accuracy} &= \frac{\text{correct conflict marks}}{\text{injected conflicts}},\\
\text{Replay Completeness} &= \frac{\text{logged replay steps}}{\text{total replay steps}}.
\end{align}

\section{Discussion}

The prototype separates event-record extraction from event-stream maintenance. Its value is in the reproducible state transition path: a record must satisfy local schema and evidence gates before it changes the versioned graph, and the resulting transition is logged. This makes controlled perturbations inspectable across local and server runs.

The limitations are central. The sample stream is deterministic and compact. The evidence gate is exact string containment. Operator prediction is rule-based. Conflict handling marks unresolved conflicts but does not implement a human review workflow. The current graph store is in memory and designed for reproducibility experiments rather than persistent graph infrastructure.

These limitations keep the claim conservative. The experiments support only the implemented simulation prototype and its controlled streams. They do not claim general extraction quality, financial utility, or deployment readiness.

\section{Conclusion}

This paper presented a reproducible versioned event graph simulation prototype for evidence-constrained financial event streams. The prototype normalizes event records, checks exact evidence spans, applies deterministic graph update operators, and replays controlled perturbation streams as a discrete-event simulation. The evaluation further includes negative-control sanity checks, scale-sensitivity results, compact LaTeX tables, and a template-based LaTeX build. The resulting evidence supports a limited but useful conclusion: controlled financial event graph updates can be replayed and audited through explicit state transition artifacts.
