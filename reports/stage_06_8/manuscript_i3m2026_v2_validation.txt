A Controlled-Replay Prototype for Auditable Versioned Financial
Event Graph Updates
Jiakai Tonga , Yue Zhaoa,∗
a School of Astronautics, Harbin Institute of Technology, Harbin, China

Abstract
Financial event records used in simulation and auditing workflows need explicit evidence links, update histories, and reproducible
replay traces. This short paper presents a controlled-replay prototype for auditable versioned financial event graph updates. The prototype accepts deterministic event records, checks required schema fields and exact evidence-span containment, applies a compact
set of graph update operators, and replays synthetic controlled streams as a discrete-event simulation. This paper does not evaluate
financial event extraction accuracy or market prediction. Instead, it studies whether controlled financial event records can be replayed into an auditable versioned event graph with explicit evidence and state-transition artifacts. The evaluation is intentionally
limited to controlled replay behavior: schema validity, evidence coverage, replay agreement under injected perturbations, replay
completeness, metadata-hidden diagnostics, a public-dataset mini-case, and lightweight runtime behavior. The results support only
a conservative auditability case-study claim for the implemented prototype, not benchmark-level generalization or deployment
performance.

1. Introduction
Financial event extraction and reporting pipelines often stop at structured records. For simulation and audit, however, the record alone is not enough: the system also needs evidence links, update histories, and a replayable statetransition trace. A duplicate record, a revised amount, or a contradictory disclosure should leave an explicit graph
transition rather than a silent overwrite.
This paper studies that narrower infrastructure problem. The contribution is a reproducible controlled-replay prototype, not a financial event extraction benchmark system and not a financial prediction system. We ask whether a
synthetic controlled stream of financial event records can be checked against source evidence, inserted into a versioned event graph, updated through a small deterministic operator set, and replayed as a discrete-event simulation
with auditable artifacts.
This work frames financial event graph maintenance as a discrete-event simulation problem: each incoming record
is an event that triggers a state transition, and the resulting version log is the simulation trace. The artifact is evaluated
as a simulation and auditability prototype rather than as an information-extraction model.
The prototype follows a design-science framing suitable for an EMSS/I3M short paper. It specifies a compact eventrecord schema, an evidence gate based on exact span containment, a versioned event graph, and a replay simulator. The
experiments use synthetic controlled streams to exercise duplicate merges, slot updates, conflict marks, and temporal
replay order. The paper reports only numbers generated by the repository scripts.
2. Related work
2.1. Financial event extraction and event records
Document-level financial event extraction systems such as DCFEE [1] and Doc2EDAG [2] motivate the need for
structured event records whose arguments may be distributed across a document. Their main problem is extraction
from financial text. Our problem begins after event-like records are available: we study whether controlled records
can be evidence-constrained, versioned, and replayed reproducibly.
2.2. Provenance, event sourcing, and versioned graph maintenance
The W3C PROV-O vocabulary [3] emphasizes explicit provenance over entities and activities. Event sourcing [4]
emphasizes reconstructing state from an event log. The prototype follows the same auditability principle in a smaller
setting: event nodes, entity links, conflicts, merges, and version logs are all kept as replay artifacts.
2.3. Discrete-event simulation for auditable event streams
Discrete-event simulation represents system evolution as event-triggered state transitions [5]. In this paper the event
stream is not a market model and does not estimate future outcomes. The simulation role is to replay controlled record
perturbations and expose each graph state transition for inspection.
3. Problem definition and simulation model
The input consists of deterministic seed records and a controlled perturbation stream. A seed record contains
an event identifier, event type, subject, object, date, trigger, evidence span, source document identifier, source text,
amount, and status. A stream record wraps an event payload with replay metadata such as arrival index, perturbation
type, base event identifier, and expected operator. This metadata is used only for controlled replay diagnostics and is
removed in the metadata-hidden check.

4. Prototype design
4.1. Evidence-constrained event records
The schema module normalizes both flat seed records and nested stream records. A record is schema-valid only
when required fields are present and the time field is an ISO date string. The evidence module then checks that the
evidence span is non-empty and appears in the source text. The gate is intentionally exact and local; it does not infer
evidence or call external services.
4.2. Versioned event graph
The graph store uses dictionary-backed event and entity nodes plus list-backed edges, conflicts, merges, and version
logs. The graph version is the length of the version log. This simple representation is sufficient for a reproducible
prototype because every accepted replay step appends an auditable record.
4.3. Update operators
The operator set is
O = {ADD EVENT, MERGE EVENT, UPDATE SLOT, MARK CONFLICT, VERSION LOG}.

(5)

ADD EVENT inserts a new event node and entity links. MERGE EVENT records that an incoming duplicate refers to
an existing event. UPDATE SLOT records changed fields and updates the target event. MARK CONFLICT appends an
unresolved conflict mark. VERSION LOG records the applied operation.
4.4. Discrete-event replay simulation
Replay orders records by arrival index. At each step the simulator runs schema validation, evidence matching,
operator prediction, target selection, graph update execution, and state logging. In the full-control replay, controlled
metadata can guide operator prediction and target selection. In the metadata-hidden diagnostic replay, perturbation
labels and direct target identifiers are removed before the decision step. Figure 1 summarizes the workflow, and

5. Experimental setup

All experiments are local or server-side script runs with fixed seeds. They do not train models, run GPU inference,
call external APIs, or copy model files. The main controlled stream starts from 30 seed samples and injects duplicate,
conflict, update, and temporal replay perturbations. Negative controls add invalid or unsupported records to test gates.
Scale sensitivity increases the seed count while keeping perturbations deterministic and proportional.

6. Results
6.1. Controlled replay and ablation
The main controlled replay experiment keeps the earlier deterministic 70-record stream. Table 2 reports the component rows and the full replay row. Component rows use “–” for non-applicable graph metrics. The full row should
be read as agreement with injected perturbation metadata under controlled conditions.
6.2. Metadata-hidden diagnostic replay
The full-control row can benefit from controlled metadata such as perturbation type, expected operator,
and base event id. To expose this dependency, the metadata-hidden replay removes perturbation labels, expected
operators, direct base-event identifiers, and gold group identifiers before operator prediction and target selection. Gold
labels are used only after replay to compute agreement. Table 3 compares the two settings.
This diagnostic is not a benchmark evaluation. If metadata-hidden agreement is lower than full-control agreement,
the result shows that controlled metadata helps the current prototype. If it remains high, the result shows only that
the deterministic rules capture part of the injected structure in the synthetic controlled stream; it does not demonstrate
general financial information-extraction performance.

6.3. Public-dataset mini-case
The public-dataset mini-case is not used as an extraction benchmark. It checks whether externally sourced financial
event records can pass the same schema, evidence, and replay pipeline used by the controlled simulation. The event
records are converted from a small subset of the open FewFC research dataset [6] and are evaluated only as replay
inputs.
Table 4 reports 20 FewFC-derived records. Schema validity, evidence coverage, and replay completeness are all
1.000 in this mini-case. These values mean only that the converted public records can pass the prototype’s local
schema, exact-evidence, and add-event replay path; they are not extraction accuracy, benchmark performance, or
evidence of general dataset coverage.
6.4. Negative-control sanity checks
The negative-control stream contains missing required fields, evidence mismatches, invalid dates, and unsupported
perturbation metadata. Table 5 reports whether the schema, evidence, date, and operator-handling paths reject or
handle these cases without crashing. This is a sanity check for gate behavior, not a broad robustness claim.
6.5. Runtime and scale sensitivity
The scale sensitivity run uses fixed seeds and controlled streams at the requested scales. Table 6 reports replay size,
active graph artifacts, replay completeness, and runtime. Figure 3 visualizes total replay runtime. Runtime can vary
across local and server environments, so the result is interpreted only as lightweight scaling behavior under controlled
streams.

7. Discussion
The prototype separates event-record extraction from event-stream maintenance. Its value is in the reproducible
state-transition path: a record must satisfy local schema and evidence gates before it changes the versioned graph, and
the resulting transition is logged. This makes controlled perturbations inspectable across local and server runs.
The limitations are central. The sample stream is deterministic and compact. The evidence gate is exact string
containment. Operator prediction is rule-based and partly assisted by controlled metadata in the full-control setting.
Conflict handling marks unresolved conflicts but does not implement a human review workflow. The current graph
store is in memory and designed for reproducibility experiments rather than persistent graph infrastructure. The public
mini-case is intentionally small and serves only as an external sanity check; larger public-dataset replay studies are
left for the camera-ready extension.

These limitations keep the claim conservative. The experiments support only the implemented controlled-replay
prototype and its synthetic controlled streams. They do not claim general extraction quality, financial utility, market
prediction, or deployment readiness.
8. Conclusion
This paper presented a controlled-replay prototype for auditable versioned financial event graph updates. The
prototype normalizes controlled financial event records, checks exact evidence spans, applies deterministic graph
update operators, and replays synthetic controlled streams as a discrete-event simulation. The evaluation includes
controlled replay agreement, a metadata-hidden diagnostic, a public-dataset mini-case, negative-control sanity checks,
scale-sensitivity results, compact LaTeX tables, and a template-based LaTeX build. The resulting evidence supports
a limited auditability case-study conclusion: controlled financial event graph updates can be replayed and inspected
through explicit state-transition artifacts in the implemented prototype.
